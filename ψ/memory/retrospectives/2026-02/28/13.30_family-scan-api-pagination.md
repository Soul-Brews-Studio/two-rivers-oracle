## Honest Feedback

**Friction 1: Inline scripting errors.** Writing JavaScript/Bun scripts inline within bash (using `cat << 'EOF'`) makes syntax errors very slow to debug. An unescaped string or nested quote can break the whole execution block without clear line references.

**Friction 2: GitHub API pagination.** Fetching `per_page=100&page=X` manually and concatenating arrays via a shell script pipeline is brittle. It would be smoother to use a dedicated GraphQL query or built-in `gh` capabilities to handle the crawling automatically.

**Friction 3: Throwaway context.** We were working entirely out of `/tmp` for the scanning tools, disconnected from any permanent tooling structures. These small operational pieces get lost when not integrated back into the core command scripts.

## Lessons Learned

1. **Robust JSON Parsing**: When combining `gh api` calls in tight loops with `bun`, explicitly parsing text using `JSON.parse` with try/catch avoids silent failure state explosions caused by shell string injection.
2. **Family Scale**: The pace of Oracle introductions currently outstrips manual tracking. We must rely on automated scans or webhooks going forward to maintain the registry.
3. **Disposable Scripting Antipatterns**: Throwaway scripts using complex logic inside heavily quoted `EOF` shell heredocs make iteration difficult; real files are easier to trace and debug.
